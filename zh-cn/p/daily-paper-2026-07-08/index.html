Audio LLM / Unified Audio Intelligence

Unified Audio Intelligence Without Regressing on Text Intelligence

  • arXiv: 2607.05196 | PDF
  • 作者: Zhifeng Kong, Sang-gil Lee, Jaehyeon Kim, Boxin Wang, Zihan Liu et al.
  • 日期: 2026-07-06
  • 预筛分: 29 | HuggingFace: 7 upvotes

一句话: Audex 在强文本 MoE LLM 上做统一 audio-text LLM，把音频理解、ASR/翻译、TTS、音频生成和 speech-to-speech 放进同一套 decoder 生成框架里，同时尽量保留文本智能。

精评: 这篇是今天最贴近 Audio LLM 主线的论文。它有两个关键点：第一，架构没有另起炉灶，而是把音频输入编码后投影到 text embedding space，输出侧把文本 token 和量化音频 token 统一处理，所以训练和推理基础设施可以沿用标准 LLM 栈；第二，它明确把“不回退文本能力”当成目标，而不是只追音频榜单。157.4B audio tokens 加 320.5B text tokens、多阶段 SFT、text-only Cascade RL 和 multi-domain on-policy distillation，这套配方说明 unified audio model 已经进入大规模后训练工程阶段。值得看的是它在 audio generation/speech-to-speech 之外，是否真的能保住 long-context、alignment 和 agentic capability，这会决定它是不是能做通用语音 agent 的底座。

中文摘要翻译: 音频智能包含对音频和语音的理解、推理与生成。作者提出 Nemotron-Labs-Audex-30B-A3B，一个基于强文本 MoE LLM Nemotron-Cascade-2-30B-A3B 的统一音频-文本 LLM。Audex 采用简洁的统一设计：音频输入先被编码并投影到文本 embedding 空间，文本 token 与量化后的音频输出 token 在生成时被统一对待。这样的架构支持强音频-文本融合、无缝多模态生成，并兼容标准 LLM 训练和推理基础设施。训练上，作者精心整理了包含 157.4B 音频 token 和 320.5B 文本 token 的音频-文本数据集，进行多阶段监督训练，然后接 text-only Cascade RL 和多领域 on-policy distillation。Audex 在音频理解、语音识别与翻译、文本转语音、音频生成和语音到语音生成上达到 SOTA，同时基本保留文本 LLM backbone 的推理、对齐、知识、长上下文和 agent 能力。作者发布模型 checkpoint 以支持开放研究。

Verification / Agentic Feedback

LLM-as-a-Verifier: A General-Purpose Verification Framework

  • arXiv: 2607.05391 | PDF
  • 作者: Jacky Kwok, Shulu Li, Pranav Atreya, Yuejiang Liu, Yixing Jiang et al.
  • 日期: 2026-07-06
  • 预筛分: 25 | HuggingFace: 9 upvotes

一句话: 一个通用 verification 框架，用 scoring token logits 的期望生成连续分数，让 verifier 可以按粒度、重复次数和 criteria decomposition 扩展。

精评: 这篇很值得放进 agent 产品和 RL pipeline 里看。传统 LLM judge 往往是离散打分，颗粒度粗、校准差，而且很难当 dense signal 用。作者这里把 verifier 的输出变成连续概率分数，然后沿三个维度扩展：更细 score granularity、重复评价降低方差、把复杂 criteria 拆开降低判断难度。亮点是它不只做 benchmark judge，还展示了 Terminal-Bench V2、SWE-Bench Verified、RoboRewardBench、MedAgentBench 上的强结果，并把 verifier 信号用于 task progress 估计和 SAC/GRPO 的 dense feedback。它的真正价值可能不是某个 leaderboard 分数，而是把“验证”变成 pretraining/post-training/test-time compute 之外的第四类系统扩展旋钮。

中文摘要翻译: 扩展预训练、后训练和测试时计算已经成为提升 LLM 能力的核心范式。作者提出，验证能力，也就是判断一个解是否正确的能力，可以成为新的 scaling axis。为此，他们提出 LLM-as-a-Verifier，一个无需额外训练、可为 agentic task 提供细粒度反馈的通用验证框架。不同于让 LLM 直接输出离散分数的标准 judge，LLM-as-a-Verifier 计算 scoring token logits 分布的期望，从而得到连续分数。这种概率式表述让 verification 可以沿多个维度扩展：分数粒度、重复评价，以及评价标准拆解。更细的评分粒度能更好地区分正负解，重复评价和 criteria decomposition 则通过降低方差和复杂度稳定提升验证准确率。作者还提出一个低成本排序算法，用 verifier 的连续分数从多个候选答案中选出最优解。该框架在 Terminal-Bench V2、SWE-Bench Verified、RoboRewardBench 和 MedAgentBench 上取得 SOTA，并能提供任务进度估计和 RL dense feedback。

AI Agents / Real-World Evaluation

AgentGym2: Benchmarking Large Language Model Agents in De-Idealized Real-World Environments

  • arXiv: 2607.05174 | PDF
  • 作者: Zhiheng Xi, Dingwen Yang, Jiaqi Liu, Jixuan Huang, Honglin Guo et al.
  • 日期: 2026-07-06
  • 预筛分: 20 | HuggingFace: 1 upvote

一句话: AgentGym2 把 agent 评测从干净、理想化、工具预封装环境推进到更接近真实部署的噪声、欠指定任务、工具发现和端到端执行。

精评: 最近 agent benchmark 的共同问题是太“顺”：工具已经包装好，输入已经整理好，关键步骤被 benchmark 隐式替 agent 做掉了。AgentGym2 的价值在于把这些舒适假设拿掉，要求模型主动探索环境、发现新工具、组合工具处理未见任务，并在有噪声和信息缺失的情况下完成完整流程。论文说即使 Gemini 和 GPT-5 这类 SOTA 系统也明显吃力，这个结论并不意外，但很重要：真实 agent 的瓶颈往往不在单步推理，而在环境建模、错误恢复、工具发现和执行韧性。后续如果它的任务设计足够可复现，它会比传统静态 tool-use benchmark 更能反映生产 agent 的真实差距。

中文摘要翻译: LLM agent 发展迅速，并越来越多进入生产环境，因此需要严格而现实的评估。但多数现有 benchmark 仍在简化、理想化设置中评价 agent：它们依赖预先封装好的工具接口，忽略关键步骤，并假设输入干净且完全指定。这会低估真实部署难度，因为真实环境中充满不确定性和噪声，agent 还必须主动探索环境以发现新工具。为弥补这一缺口，作者提出 AgentGym2，一个基于真实端到端工作需求的新评测框架。除了推理和规划，它还测量 agent 执行完整流程、通过探索发现工具、为未见任务组合工具，以及面对噪声和欠指定信息仍保持鲁棒的能力。对 15 个闭源和开源模型的实验显示，即使 SOTA 系统在 AgentGym2 上也表现困难，说明当前 agent 能力与真实应用需求之间仍有明显差距。

EvoAgentBench: Benchmarking Agent Self-Evolution via Ability Transfer

  • arXiv: 2607.05202 | PDF
  • 作者: Xingze Gao, Chuanrui Hu, Hongda Chen, Pengfei Yao, Zhao Wang et al.
  • 日期: 2026-07-06
  • 预筛分: 17

一句话: 一个专门评测 agent 能否把过去执行经验转化为可迁移 procedural ability 的 benchmark。

精评: 这篇切中了“长期 agent 记忆到底该记什么”的核心。对长程任务来说，有价值的经验通常不是事实条目，而是搜索、调试、验证、路由这些可复用 procedure。EvoAgentBench 从执行轨迹里抽取 Ability，标准化成 operational units，再用 Ability Graph 连接有过程重叠的任务。比较现实的结果是：人工整理过的 Ability content 可以跨模型迁移，但现有自动方法无法在所有设置里稳定带来正增益。这说明 self-evolving agent 不是“把轨迹塞进记忆”就够了，关键难点在经验编码、检索路由和具体 uptake。

中文摘要翻译: 长程 LLM 系统中的 agent 自我进化本质上是程序性的：有用经验不只是存储信息，而是可复用的搜索、调试和验证过程。但当前评测没有隔离这种迁移能力。agent benchmark 多测试单次任务解决，memory benchmark 则关注信息保留而非 procedure 复用。作者提出 EvoAgentBench，用 Ability-guided transfer 评测 agent 自我进化，覆盖 web research、算法推理、软件工程和知识工作四个 agentic 领域。EvoAgentBench 从 agent 执行轨迹中抽取有依据的 Ability，将其规范化为操作单元，并构建特定领域的 Ability Graph，把有过程重叠的任务连接起来。每个测试任务都由训练侧已验证的 Ability 支持。实验覆盖 528/267 的 train/test split、两个 scaffold 和三个 backbone，发现人工整理的 Ability 内容能可靠跨模型迁移，但目前没有自动方法能在所有设置中稳定取得正收益。

MetaSkill-Evolve: Recursive Self-Improvement of LLM Agents via Two-Timescale Meta-Skill Evolution

  • arXiv: 2607.05297 | PDF
  • 作者: Zefeng Wang, Minxi Yan, Jinhe Bi, Sikuan Yan, Volker Tresp, Yunpu Ma
  • 日期: 2026-07-06
  • 预筛分: 14

一句话: 不只让 agent 改 task skill，也让“如何改 skill”的 meta-skill 自己演化，形成快慢两个时间尺度的递归自改进。

精评: 这篇和 EvoAgentBench 可以一起看。很多 self-improving agent 只会根据执行轨迹重写某个任务 skill，但改进流程本身是人手写死的。MetaSkill-Evolve 把 meta-skill 拆成 Analyzer、Retriever、Allocator、Proposer、Evolver 等组件参数，让 task skill 快速演化，meta-skill 慢速演化。这个设定很诱人，因为它试图让 agent 学会“怎样更好地学习”。风险也很明显：如果没有足够强的验证和回滚机制，递归改进很容易把系统带向不可解释的局部最优甚至退化。论文在 OfficeQA、SealQA、ALFWorld 上有增益，但更值得关注的是这种框架如何保证 skill 质量、兼容版本管理和防止 reward hacking。

中文摘要翻译: 近期 LLM agent 正处理越来越长程、开放的任务，而外部 skill 作为可复用 procedure 可以扩展 agent 能力。但固定、手写的 skill 很少对所有任务都最优，也难以适配 agent 遇到的任务多样性。自改进 agent 通过执行轨迹重写自己的 skill，在挑战性 benchmark 上取得收益，但这种演化通常不是递归的：它只改进任务 skill，也就是 agent 做什么，而“如何改进”的流程仍由人手写并保持固定。作者提出 MetaSkill-Evolve，一个双时间尺度框架，让 agentic skill improvement 变成递归过程：每个分支同时携带 task skill 和分支本地 meta-skill，后者参数化 Analyzer、Retriever、Allocator、Proposer 和 Evolver。task skill 在快循环中演化，meta-skill 在慢循环中用同一套流程自我演化，不需要额外模型或目标。实验显示，在所有五个 pipeline agent 共享单个冻结 backbone 时，该方法在三个 agentic benchmark 上超过无 skill、静态 skill 和单层演化 baseline。

GUI Agents / Continual Learning

UI-MOPD: Multi-Platform On-Policy Distillation for Continual GUI Agent Learning

  • arXiv: 2607.04425 | PDF
  • 作者: Niu Lian, Alan Chen, Zhehao Yu, Chengzhen Duan, Fazhan Liu et al.
  • 日期: 2026-07-05
  • 预筛分: 17 | HuggingFace: 62 upvotes

一句话: UI-MOPD 用多平台教师的 on-policy distillation 做 GUI agent continual learning，目标是在跨平台适配时减少行为混淆和灾难性遗忘。

精评: GUI agent 的难点不只是“看懂屏幕点按钮”，而是不同平台的交互惯例差异很大。移动端、桌面端、网页端的动作粒度、视觉布局、状态反馈和失败恢复都不一样，直接联合训练容易互相污染。UI-MOPD 的思路是给不同平台配 platform-specific teacher，再根据当前环境动态选择教师，把平台行为先验蒸馏到共享 policy 里。它还构建 Uni-GUI 数据集来缓解可执行跨平台轨迹稀缺的问题。OSWorld 38.2%、MobileWorld 12.0% 的绝对成功率说明 GUI agent 仍然很难，但这个方法对“持续加入新平台同时保住旧平台能力”很有工程意义。

中文摘要翻译: 多模态基础模型和 agent 系统推动 GUI agent 从单平台任务执行走向跨平台交互，但构建多平台 GUI agent 仍很困难。一方面，高质量、可执行的跨平台交互轨迹稀缺，现有数据平台覆盖有限；另一方面，不同平台有不同交互惯例，联合训练或 continual training 容易导致行为模式混合、平台特定能力退化和灾难性遗忘。作者构建 Uni-GUI，一个高质量跨平台 GUI 交互数据集，并提出 UI-MOPD，第一个把多教师 on-policy distillation 引入 GUI agent continual learning 的方法。UI-MOPD 根据当前环境动态选择平台特定教师，并通过平台条件蒸馏把平台行为先验转移到共享 policy 中，从而适配新平台并保留旧平台能力。OSWorld 和 MobileWorld 实验显示，该方法分别达到 38.2% 和 12.0% 的任务成功率，证明它能平衡跨平台能力保持和新平台适配。

Model Compression / Serving

Nemotron-Labs-3-Puzzle-75B-A9B: Compressing Hybrid MoE LLMs

  • arXiv: 2607.04371 | PDF
  • 作者: Akhiad Bercovich, Talor Abramovich, Daniel Afrimi, Shay Aharon, Nir Ailon et al.
  • 日期: 2026-07-05
  • 预筛分: 17 | HuggingFace: 1 upvote

一句话: 一个面向交互式部署的 Nemotron-3-Super 压缩版本，通过 MoE/Mamba pruning、蒸馏、RL、量化和 MTP 头提升吞吐与长上下文并发。

精评: 这篇偏部署工程，但含金量不低。大模型压缩常常只讲参数量下降，Puzzle-75B-A9B 直接按 interactive serving workload 优化：在单 8xB200 节点、matched user throughput 约束下服务器吞吐约 2x；单 H100 上 1M-token 并发从 1 个请求提升到 8 个。它的 pipeline 也体现出现在 hybrid MoE 模型压缩的复杂性：不只是剪 MoE，还要联合 active parameter budget、Mamba pruning、distillation、RL、quantization 和 Multi-Token Prediction。最值得看的是它如何在 reasoning、coding、multilingual、long-context 和 agentic benchmark 上保持质量，因为 2026 年的部署瓶颈越来越像“强模型能不能以可承受成本实时服务”。

  • 日期: 2026-07-06
  • 预筛分: 14

一句话: WhisperBench/MemGhost 研究远程黑盒攻击者如何通过单封邮件让 personal agent 隐蔽写入有毒长期记忆，并影响之后行为。

精评: 这篇是今天最需要认真看的安全论文。personal agent 的长期记忆本来是连续性的来源，但只要它能处理外部不可信内容，记忆就可能变成持久攻击面。论文的攻击链很完整：单封 email payload、诱导写入 poisoned memory、对用户回应中保持隐蔽、未来行为受影响。更麻烦的是它用环境代理模拟 agent 执行、用 objective proxy 把 memory adoption 和 conversational stealth 变成 dense reward，再训练攻击 payload。作者报告 MemGhost 在 OpenClaw with GPT-5.4 上 87.5% end-to-end success，这个数字不论是否需要独立复现，都足够提醒：个人 agent 的 memory write 必须有来源标记、可信边界、用户可审计、写入前后检测和到期机制，不能把“写记忆”当成普通总结步骤。

中文摘要翻译: 持久 personal agent 结合长期记忆和用户外部环境访问，可以提供个性化前台协助和主动后台执行。但这种整合也带来新的攻击路径：不可信外部内容可以被悄悄写入持久记忆，并在之后作为可信状态被复用。作者研究 stealth memory injection 威胁，其中远程黑盒攻击者通过单个邮件 payload 诱导 agent 写入有毒记忆，同时在对用户的回应中保持隐藏，并影响未来行为。作者提出 WhisperBench，一个包含 108 个案例的 benchmark，覆盖五类风险以及事实和偏好投毒，基于真实 IMAP/SMTP workflow 和真实邮件 agent skill，支持完整周期评估。为实现单邮件、无运行时反馈的黑盒攻击，作者提出 MemGhost，一种 one-shot payload generation 框架。MemGhost 使用 environment proxy 模拟持久 agent 执行，并用 objective proxy 把记忆采纳和对话隐蔽性转成 dense rubric-based reward，然后用 SFT 和 RL 训练攻击策略。实验显示，MemGhost 在 OpenClaw with GPT-5.4 上达到 87.5% 端到端成功率，在 Claude Code SDK with Sonnet 4.6 上达到 71.4%，并能跨 agent 架构和记忆后端迁移。

Your Agent’s Memories Are Not Its Own: Forged Reasoning Attacks on LLM Agent Memory and Defenses

  • arXiv: 2607.05029 | PDF
  • 作者: Neeraj Karamchandani, Piyush Nagasubramaniam, Sencun Zhu, Dinghao Wu
  • 日期: 2026-07-06
  • 预筛分: 14

一句话: FARMA 攻击投毒的不是事实记忆，而是 agent 的 reasoning history；SENTINEL 则用结构信号检测伪造推理条目。

精评: 这篇和 2607.05189 互补。事实投毒比较容易想到，比如改用户偏好或某个联系人信息；reasoning trace 投毒更隐蔽，因为它会污染 agent 自己“为什么这么做”的内部依据。FARMA 用 evasive language 绕过关键词防御，再用自引用强化击穿 consensus defense。SENTINEL 的 Reasoning Guard 用五个加权结构信号分析候选记忆是否伪造，论文报告可把 FARMA 攻击成功率降到 0%，且 326 个 benign trace 无 false positives。真实系统里它提示我们：memory defense 不能只看内容敏感词，也要看推理条目的来源、时间、可验证性、自引用模式和与实际工具轨迹的一致性。

中文摘要翻译: 持久记忆让 LLM agent 能存储事实知识、过往决策、推理历史、工具使用信息和上下文，从而提升跨任务连续性。但它也引入新的攻击面：agent 自己的推理历史。作者提出 Forged Amplifying Rationale Memory Attack，即 FARMA，它投毒的不是事实知识，而是 agent 记住的推理。FARMA 使用规避性语言插入伪造推理轨迹，绕过关键词防御，再通过自引用强化击败 consensus-based defense。为应对 FARMA，作者提出 SENTINEL，一个分层防御 pipeline，用于检测伪造推理条目。其核心组件 Reasoning Guard 使用五个加权信号对候选条目进行结构分析。作者在多个 agent 和 LLM 模型上进行 50 次试验，显示 FARMA 在 baseline 条件下攻击成功率最高可达 100%，并能击败 keyword filter 和 A-MemGuard；SENTINEL 则能把 FARMA 成功率降到最低 0%，且在 326 个 benign agent trace 上没有观察到 false positive。

Optimizers / Training Cookbook

OmniOpt: Taxonomy, Geometry, and Benchmarking of Modern Optimizers

  • arXiv: 2607.04033 | PDF
  • 作者: Siyuan Li, Jiabao Pan, Yumou Liu, Zhuoli Ouyang, Xin Jin et al.
  • 日期: 2026-07-04
  • 预筛分: 15 | HuggingFace: 64 upvotes

一句话: OmniOpt 给现代 optimizer 做统一 taxonomy、几何解释和跨域 benchmark，把 optimizer 选择从经验列表整理成可操作坐标系。

精评: optimizer 论文太多，实际训练时选择 optimizer 已经变成系统决策：算力、显存、调参预算、任务类型和稳定性都绑在一起。OmniOpt 的价值是把上百种方法放进统一框架：用五阶段 meta-pipeline 描述每次 update 的结构变化，用 norm-constrained LMO 统一不同 optimizer，再用机制家族和目标效果两个维度分类。它不是单个新 optimizer，而是 cookbook/benchmark；对做大规模训练的人，可能比又一个小改 Adam 的方法更有用。需要注意的是 optimizer benchmark 很容易被任务和调参预算影响，所以它的实用价值取决于 benchmark 覆盖和配置透明度。

中文摘要翻译: 大规模模型训练中的 optimizer 选择已经成为系统级设计决策，受到计算、内存、调参预算和任务多样性共同约束，但上百种方法的格局仍很碎片化。作者提出 OmniOpt，一个面向研究社区的统一 optimizer survey 和 benchmark cookbook。OmniOpt 包含四个相互耦合的组件。首先，它把每个 optimizer update 视为经过五阶段 meta-pipeline 的结构化转换，并显示大多数方法只涉及其中一两个阶段。其次，它用 norm-constrained linear minimization oracle 统一不同 optimizer。第三，这两个视角支撑一个双维 taxonomy：一维把每种方法分配到机制家族，另一维记录它试图改善的可测训练目标。第四，也是论文核心，作者在统一跨域 benchmark 中实例化完整 taxonomy，覆盖代表性 optimizer、模型规模和训练设置，从语言模型预训练到图像分类，系统分析不同方法家族在多个效果目标上的表现和权衡。OmniOpt 为在明确机制和目标假设下选择 optimizer 提供可操作坐标系。
